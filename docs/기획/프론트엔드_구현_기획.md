# Pocketman 프론트엔드 구현 기획

> 작성일: 2026-03-11
> 기준: `docs/기획/프로젝트_기획안_v5_최종확정.md` + `docs/common_rule/frontend_context.md` + `docs/develop_rule/`
> 목표: 3주 MVP — 데스크톱 퍼스트 반응형, Instagram/Facebook 구조 모델 적용

---

## 1. 설계 철학

### 1-1. 레퍼런스 모델: Instagram/Facebook 구조

| 구성 요소 | Instagram | Pocketman 적용 |
|-----------|-----------|----------------|
| 상단 고정 헤드 | 로고 + 검색 + 알림 + 프로필 | 로고 + 광장 이동 + 새 크리처 만들기 CTA |
| 좌측 사이드바 | 네비게이션 메뉴 | (lg 이상) 사이드 네비: 홈, 광장, 크리처 만들기 |
| 중앙 피드 | 게시물 피드 | 광장 피드 (public creatures) |
| 우측 사이드바 | 추천 계정 / 광고 | (xl 이상) 인기 포켓몬 랭킹 (선택 구현) |
| 개별 상세 | 게시물 모달 | 크리처 상세 모달 or `/creatures/[id]` 페이지 |
| 스토리 | 상단 스토리 롤 | 인트로 페이지 (온보딩 애니메이션) |

### 1-2. 핵심 원칙

1. **페이지 단위 라우팅**: 단일 파일에 useState step 스위치 구조를 제거하고, 각 단계를 독립 페이지로 분리한다
2. **URL이 상태**: 현재 사용자의 위치와 진행 상태는 URL로 표현되어 새로고침 시 복구 가능해야 한다
3. **컴포넌트 모듈화**: 400줄 초과 금지 (`docs/develop_rule/01_global_rules.md` §2.4) — 모든 UI는 재사용 가능한 컴포넌트 단위로 분리
4. **API 분리 원칙**: API 호출은 반드시 `lib/api.ts`에만 작성한다. 컴포넌트/페이지에 fetch 직접 작성 금지
5. **타입 중앙화**: `lib/types.ts`에 없는 타입을 인라인으로 정의하지 않는다

## 2. 페이지 라우팅 구조

### 2-1. 전체 라우트 맵

```
frontend/app/
│
├── page.tsx                     # / — 랜딩 페이지 (마케팅 CTA)
├── intro/
│   └── page.tsx                 # /intro — 인트로 애니메이션 (온보딩)
├── upload/
│   └── page.tsx                 # /upload — 사진 업로드
├── match/
│   └── page.tsx                 # /match — Top-3 포켓몬 선택
├── generate/
│   └── [id]/
│       └── page.tsx             # /generate/[id] — 생성 진행 중 (로딩/폴링)
├── result/
│   └── [id]/
│       └── page.tsx             # /result/[id] — 크리처 프로필 완성 화면
├── creatures/
│   └── [id]/
│       └── page.tsx             # /creatures/[id] — 개별 크리처 상세 (공유 링크용)
├── plaza/
│   └── page.tsx                 # /plaza — SNS 광장 피드
│
├── layout.tsx                   # 루트 레이아웃 (헤드 + 사이드바 포함)
└── globals.css                  # 전역 스타일 + 애니메이션
```

### 2-2. 사용자 플로우 (페이지 전환 흐름)

```
[/] 랜딩
  └─ CTA 클릭 "포켓몬 찾기 시작" ──▶ [/intro] 인트로 애니메이션
                                          └─ 애니메이션 완료 ──▶ [/upload] 사진 업로드
                                                                    └─ 업로드 + 분석 ──▶ [/match] Top-3 선택
                                                                                          └─ 포켓몬 선택 ──▶ [/generate/{id}] 생성 진행
                                                                                                              └─ 생성 완료 ──▶ [/result/{id}] 결과 화면
                                                                                                                                ├─ 광장 등록 ──▶ [/plaza] SNS 피드
                                                                                                                                └─ 공유 링크 ──▶ [/creatures/{id}] 개별 상세
[/plaza] SNS 피드
  └─ 크리처 카드 클릭 ──▶ [/creatures/{id}] 개별 상세 (모달 or 페이지)
```

### 2-3. 페이지 간 데이터 전달 전략

> URL 파라미터 + sessionStorage 조합. 전역 상태관리(Redux/Zustand) 미사용.

| 구간 | 전달 방식 | 전달 데이터 |
|------|-----------|------------|
| `/upload` → `/match` | `sessionStorage` | `matchResult` (Top-3 데이터, user_vector) |
| `/match` → `/generate/[id]` | URL 파라미터 `id` | `creature_id` (백엔드에서 생성된 creature UUID) |
| `/generate/[id]` → `/result/[id]` | URL 파라미터 `id` | 동일 `creature_id` |
| `/result/[id]` → `/plaza` | URL `?focus={id}` | 하이라이트 대상 creature_id |
| 공유 링크 | URL `/creatures/{id}` | creature_id |

```
sessionStorage 키 정의:
- "pm_match_result"  : JSON(MatchResponse)  ← /upload에서 저장, /match에서 읽기
- "pm_upload_image"  : base64 or objectURL  ← 미리보기용 (선택)
```

### 4-2. `/intro` 인트로 페이지

**목적**: 서비스 플로우 소개 + 몰입감 연출

**시퀀스 애니메이션 (3~5초)**:
```
Step 1 (1s): "당신의 얼굴을 분석합니다" + 얼굴 스캔 아이콘 애니메이션
Step 2 (1s): "포켓몬 386마리와 비교합니다" + 포켓몬 카드 플립 애니메이션
Step 3 (1s): "당신만의 크리처가 탄생합니다" + 크리처 reveal 애니메이션
Step 4 (0.5s): "지금 시작하세요" + 자동 → /upload 이동 또는 버튼 클릭
```

**구현 포인트**:
- `IntroAnimation.tsx` 컴포넌트에서 useState로 step 관리
- `useEffect` + `setTimeout`으로 자동 전진
- 스킵 버튼 제공 ("건너뛰기")
- 완료 후 `router.push('/upload')`

### 4-3. `/upload` 업로드 페이지

**목적**: 사진 업로드 + 클라이언트 사전 검증

**레이아웃**:
```
┌────────────────────────────────────────────────────┐
│  HEADER                                            │
├──────────────────────────┬─────────────────────────┤
│  좌측: 업로드 영역        │  우측: 안내 텍스트       │
│  [드래그앤드롭 박스]      │  - 정면 사진 권장        │
│  또는 [파일 선택] 버튼    │  - 밝은 조명             │
│                          │  - JPG/PNG/WebP, 10MB   │
│  [미리보기 이미지]        │                         │
│                          │  [분석 시작] 버튼 (활성) │
└──────────────────────────┴─────────────────────────┘
```

**클라이언트 검증 (API 호출 전)**:
- 파일 크기: 10MB 이하
- MIME 타입: image/jpeg | image/png | image/webp
- 최소 해상도: 100×100px (FileReader + Image 객체로 검사)

**상태 흐름**:
```
idle → file_selected → validating → uploading → (redirect to /match)
```

**오류 처리**:
- 검증 실패: Toast로 오류 메시지 표시, idle로 복귀
- API 실패: "얼굴을 찾지 못했습니다. 다른 사진을 사용해주세요" + 재시도 버튼
- 네트워크 오류: "네트워크 오류가 발생했습니다" + 재시도 버튼

**API 호출 완료 후**:
```typescript
sessionStorage.setItem("pm_match_result", JSON.stringify(matchResult))
router.push("/match")
```

**데이터 소스**: `sessionStorage.getItem("pm_match_result")` 파싱

**선택 후 흐름**:
```typescript
// 1. creature 생성
const creature = await createCreature({ pokemon_id, match_rank, ... })
// 2. 생성 파이프라인 시작
await generateCreature(creature.id, { trigger_video: true })
// 3. generate 페이지로 이동
router.push(`/generate/${creature.id}`)
```

**useVeoPolling 훅**:
```typescript
// hooks/useVeoPolling.ts
function useVeoPolling(jobId: string | null) {
  // MAX=25, INTERVAL=3000ms
  // 완료: status="succeeded" → video_url 반환
  // 타임아웃: 25회 초과 → null 반환 (영상 없이 result 진행)
  // 실패: status="failed" → null 반환
}
```

**생성 완료 조건**:
- 이미지 + 스토리 완료: `creature.image_url != null && creature.story != null`
- 영상 완료(선택): `veoJob.status === "succeeded"`
- 영상 타임아웃: 25회 폴링 초과 → 영상 없이 result로 이동

```typescript
// 생성 완료 후
router.push(`/result/${id}`)
```

## 6. 상태관리 전략

### 6-1. 원칙

- **전역 상태 미사용**: Redux/Zustand 불필요. 각 페이지가 자기 상태를 관리
- **URL이 진실의 단일 원천**: 어떤 creature를 보는지는 `[id]` URL 파라미터로 결정
- **sessionStorage는 임시 전달용**: 페이지 간 이동 시 1회성 데이터 전달 (브라우저 닫으면 삭제)

### 6-2. 상태 범위별 분류

| 데이터 | 저장소 | 이유 |
|--------|--------|------|
| 현재 페이지 | URL | Next.js App Router |
| 매칭 결과 (Top-3) | `sessionStorage["pm_match_result"]` | 페이지 이동 후 사용, 새로고침 불필요 |
| 업로드 이미지 미리보기 | `sessionStorage["pm_upload_preview"]` | 선택적, 용량 주의 |
| 크리처 데이터 | `GET /creatures/{id}` 직접 fetch | 항상 최신 데이터 |
| 리액션 카운트 | 로컬 useState + 낙관적 업데이트 | 즉각적 UX |
| BGM 재생 상태 | `localStorage["pm_bgm_on"]` | 브라우저 닫아도 유지 |

### 6-3. storage.ts 헬퍼

```typescript
// lib/storage.ts
export const MatchResultStorage = {
  save: (data: MatchResponse) =>
    sessionStorage.setItem("pm_match_result", JSON.stringify(data)),
  load: (): MatchResponse | null => {
    const raw = sessionStorage.getItem("pm_match_result")
    return raw ? JSON.parse(raw) : null
  },
  clear: () => sessionStorage.removeItem("pm_match_result"),
}
```
