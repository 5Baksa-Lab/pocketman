# Stage 문서 작성 플레이북

## 1) Stage 문서 필수 섹션

모든 Stage 문서는 아래 10개 섹션을 동일하게 유지한다.

1. Stage 목표
2. 포함 라우트 / 제외 라우트
3. 사용자 플로우
4. 페이지별 UI 구현 단계
5. 공통 컴포넌트/모듈 계획
6. 상태 전이 모델
7. API 계약 요약
8. 예외/엣지케이스
9. QA 체크포인트
10. 후속 백로그

---

## 2) 페이지 구현 단계 기술 방식

각 페이지는 반드시 아래 구조로 작성한다.

- `목표`
- `UI 구조 (Desktop/Tablet/Mobile)`
- `상태`
- `구현 단계(순차)`
- `완료 기준`

예시 포맷:

```text
### /example

목표:
- ...

UI 구조:
- Desktop: ...
- Mobile: ...

상태:
- idle/loading/success/error/empty

구현 단계:
1. ...
2. ...
3. ...

완료 기준:
- ...
```

---

## 3) 상태 전이 모델 작성 규칙

상태 전이표는 최소 아래 열을 포함한다.

- `현재 상태`
- `트리거`
- `다음 상태`
- `사용자 피드백`
- `부수효과(API/스토리지/라우팅)`

---

## 4) API 계약 표 작성 규칙

엔드포인트별로 다음을 문서화한다.

- `Method + Path`
- `요청 파라미터`
- `성공 응답 최소 필드`
- `실패 코드 및 사용자 메시지`
- `재시도 가능 여부`

---

## 5) 예외/엣지케이스 범주

각 Stage 문서에서 최소 다음 범주를 다룬다.

1. 네트워크 지연/실패
2. 권한(비로그인/만료세션)
3. 빈 데이터
4. 유효하지 않은 URL 파라미터/ID
5. 멱등성/중복 클릭
6. 브라우저 제약(Web Share, autoplay, storage)

---

## 6) QA 체크포인트 작성 규칙

QA는 반드시 아래 3그룹으로 나눈다.

1. Happy path
2. Failure path
3. Recovery path

각 시나리오는 다음 형식으로 작성한다.

- `사전조건`
- `행동`
- `기대결과`

---

## 7) 후속 백로그 작성 규칙

현재 Stage에서 구현하지 않는 항목은 아래로만 기록한다.

- `항목명`
- `미루는 이유`
- `예상 배치 Stage`
- `선행조건`

이 규칙을 지키면 문서가 구현 계획이 아닌 아이디어 메모로 흐려지는 것을 방지할 수 있다.
