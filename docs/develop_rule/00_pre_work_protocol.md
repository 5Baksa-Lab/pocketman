# 작업 프로토콜 (Work Protocol)

> **이 규칙은 반드시 지켜져야 한다.**
> 어떤 파트, 어떤 Stage의 작업이든 예외 없이 아래 순서를 따른다.

---

## 전체 흐름

```
[착수 전]
Step 1. AI가 계획 보고
           ↓
Step 2. 사용자: "동의합니다." 답변
           ↓
Step 3. AI가 필수 질문 3가지 제시
           ↓
Step 4. 사용자가 3가지 답변
           ↓
[작업]
Step 5. 코드 작업 수행
           ↓
[착수 후]
Step 6. 빌드 테스트 통과 확인  ← 실패 시 코드로 돌아가 수정
           ↓
Step 7. 개발일지 작성
           ↓
Step 8. 사용자가 코드 리뷰 진행
           ↓
Step 9. 리뷰 리스크 수정 + 사용자 확인
           ↓
        다음 작업 착수 (Step 1로 반복)
```

---

## Step 1 — 작업 전 계획 보고

작업을 시작하기 전에 반드시 아래 4가지를 사용자에게 먼저 설명한다.

1. **현재 단계**: 어느 Phase / Stage를 시작하는지
2. **작업 내용**: 구체적으로 무엇을 할 것인지
3. **이유**: 왜 이 작업이 지금 필요한지
4. **최선 여부**: 이 접근 방식이 최선인 근거 (다른 선택지와 비교)

보고 형식 예시:
```
[작업 계획 보고]

Phase: Phase 1 — Critical Bug Fix
작업: 1-A. cv_adapter.py 임시파일 누수 수정

작업 내용:
  backend/app/adapter/cv_adapter.py의 os.unlink() 호출을
  try/finally 블록으로 이동하여 예외 발생 시에도 임시파일이
  반드시 삭제되도록 수정합니다.

이유:
  현재 extractor.extract() 도중 예외가 발생하면 임시파일이
  삭제되지 않고 누적되어 서버 디스크 고갈을 유발합니다.

이 접근이 최선인 이유:
  try/finally는 Python 표준 패턴으로 예외 여부와 무관하게
  finally 블록을 보장합니다. tempfile.NamedTemporaryFile의
  delete=True 옵션은 Windows 환경 호환 문제가 있어 제외했습니다.
```

---

## Step 2 — 사용자 동의 확인

계획을 보고한 뒤, **사용자로부터 "동의합니다."라는 답을 받을 때까지 작업을 진행하지 않는다.**

- 사용자가 수정이나 질문을 하면 계획을 조정한 후 다시 보고한다.
- "동의합니다." 외의 답변은 동의로 간주하지 않는다.

---

## Step 3 — 착수 전 필수 질문 3가지

"동의합니다."를 받은 직후, 작업을 시작하기 전에 반드시 아래 3가지를 사용자에게 질문한다.

```
[착수 전 필수 확인 사항]

Q1. 현재 작업 중인 브랜치와 로컬 환경 상태가 깨끗한가요?
    (미커밋 변경사항, 충돌 파일 등이 없는지)

Q2. 이 작업과 연관된 다른 팀원의 진행 중인 작업이 있나요?
    (같은 파일을 동시에 수정 중인 경우 충돌 가능)

Q3. 지금 이 작업 외에 함께 처리하길 원하는 추가 요구사항이 있나요?
    (범위를 미리 확정해야 작업 도중 방향이 바뀌는 것을 방지)
```

3가지 질문에 대한 답을 모두 받은 후에만 실제 코드 작업을 시작한다.

---

## Step 4 — 작업 수행

Step 3의 답변을 반영하여 코드 작업을 수행한다.
작업 중 범위를 벗어나는 수정이 필요하면 **작업을 멈추고 사용자에게 보고한 후 판단을 받는다.**

---

## Step 5 — 빌드 테스트 (작업 완료 판정 기준)

**코드 수정 후 반드시 빌드/실행 테스트를 통과해야 작업 완료로 인정한다.**
빌드 테스트 실패 상태에서는 개발일지를 작성하거나 다음 단계로 넘어가지 않는다.

### 파트별 빌드 테스트 기준

**Backend**
```bash
# 의존성 설치 확인
pip install -r backend/requirements.txt

# 문법 오류 없이 임포트 가능한지 확인
cd backend && python -c "from app.main import app; print('OK')"

# 서버 기동 확인 (USE_MOCK_AI=true)
USE_MOCK_AI=true uvicorn app.main:app --port 8000
# → GET http://localhost:8000/health 응답 200 확인
```

**Frontend**
```bash
cd frontend
npm install
npm run build   # 빌드 오류 없어야 함
npm run lint    # ESLint 오류 없어야 함
```

**MLOps 스크립트**
```bash
# 변경된 스크립트 단독 실행 가능 여부 확인
python3 scripts/<변경된_스크립트>.py --help
```

빌드 테스트 결과는 개발일지의 3) 테스트/검증 항목에 반드시 기록한다.

---

## Step 6 — 개발일지 작성 (빌드 테스트 통과 후)

**빌드 테스트를 통과한 후 반드시 해당 작업과 관련된 개발일지를 작성한다.**
다음 작업을 시작하기 전에 개발일지가 존재해야 한다.

### 개발일지 경로 규칙
```
docs/development_logs/<파트>/stage-<stage>/DEVLOG.md
```

예시:
- `docs/development_logs/backend/stage-b1-core-mock/DEVLOG.md`
- `docs/development_logs/frontend/stage-f2-result-generation/DEVLOG.md`

### 개발일지 생성 방법
```bash
bash docs/develop_rule/scripts/init_stage_devlog.sh \
  --part <파트> \
  --stage <stage-id> \
  --owner "<이름>"
```

### 개발일지 필수 작성 항목
`docs/develop_rule/templates/DEVLOG_TEMPLATE.md` 기준으로 아래 항목을 반드시 채운다.

1. 작업 목표 및 범위
2. 구현 상세 (핵심 로직, 의사결정 이유)
3. 변경 파일 목록
4. 테스트/검증 결과 (USE_MOCK_AI=true 여부 명시)
5. 이슈/리스크/미해결 항목
6. 다음 작업 인수인계 내용
7. AI 활용 기록

---

## Step 7 — 사용자 코드 리뷰

**개발일지 작성 후, 사용자에게 코드 리뷰를 요청한다.**
코드 리뷰가 완료되기 전까지 다음 작업을 시작하지 않는다.

AI는 리뷰 요청 시 아래 항목을 함께 제공한다.

```
[코드 리뷰 요청]

변경 파일 목록:
  - backend/app/adapter/cv_adapter.py (os.unlink try/finally 이동)

핵심 변경 내용:
  (변경 전/후 코드 비교 또는 요약)

확인 요청 사항:
  - 로직이 의도대로 동작하는지
  - 예상치 못한 부작용이 있는지
  - 더 나은 접근 방식이 있는지

잔여 리스크:
  (이번 작업 후 남아 있는 알려진 리스크)

개발일지 경로:
  docs/development_logs/backend/stage-.../DEVLOG.md
```

---

## Step 8 — 리뷰 리스크 수정

사용자의 코드 리뷰에서 발견된 리스크나 수정 요청이 있으면:

1. 수정 사항을 파악하고 범위를 확인한다.
2. 수정 범위가 크면 Step 1~3을 다시 거쳐 동의를 받고 진행한다.
3. 수정 범위가 작으면 (오타, 변수명, 명백한 버그) 바로 수정한다.
4. 수정 후 개발일지를 업데이트한다.

**모든 리스크가 해결된 것을 사용자가 확인한 후에만 다음 작업으로 이동한다.**

---

## 이 프로토콜을 건너뛰어도 되는 경우

없다. 예외 없이 적용한다.

단, 아래는 "작업"으로 간주하지 않으므로 프로토콜 없이 수행 가능하다.
- 파일/코드 읽기 및 분석
- 질문에 대한 답변
- 문서 내용 확인
