# PM 비판적 리스크 분석 리포트 (기획안 v2 대상)

**작성일:** 2026년 3월 5일
**작성자:** 10년 차 시니어 PM / 서비스 기획자
**문서 목적:** Pokéman 기획안 v2(멀티모달 SNS)의 실현 가능성 및 잠재적 실패 요인에 대한 냉정한 비판과 현실 점검

---

## 1. 아키텍처 및 파이프라인의 비대화 (Over-engineering)
가장 큰 문제는 **3주라는 기간 대비 스코프가 기형적으로 크다**는 점입니다. '바이브 코딩'이 코드를 빨리 짜게 해줄 수는 있어도, 복잡한 파이프라인의 아키텍처적 결함까지 알아서 해결해주지는 않습니다.

*   **비판 지점:** 현재 파이프라인은 `[사용자 사진] $\rightarrow$ [CV 분석] $\rightarrow$ [LLM 텍스트 생성] $\rightarrow$ [GenAI 이미지 생성] $\rightarrow$ [TTS 음성 생성] $\rightarrow$ [FFmpeg 영상 인코딩] $\rightarrow$ [AWS/GCP 스토리지 업로드] $\rightarrow$ [DB 저장 및 클라이언트 반환]` 이라는 8단계를 거칩니다. 
*   **리스크:** 이 중 단 하나의 API 연결이나 서버에서 병목(Bottleneck)이 발생하거나 Time-out이 나면, 전체 생성 프로세스가 실패합니다. 에러 핸들링과 비동기(Async) 큐(예: Celery, Redis) 처리를 완벽하게 구축하지 않으면 데모 시연조차 불가능할 수 있습니다.

## 2. 레이턴시(Latency)와 사용자 경험(UX) 붕괴
기획안은 "빠른 응답 속도를 위해 정지 이미지+FFmpeg를 쓴다"고 타협했지만, 여전히 무겁습니다.

*   **비판 지점:** 이미지 생성(5~10초) + 텍스트 생성(3~5초) + TTS(2~3초) + 백엔드 FFmpeg 영상 렌더링(최소 10~20초) = **유저는 결과물을 보기 위해 최소 20초에서 40초를 대기해야 합니다.**
*   **리스크:** 현대 SNS에서 프로필 하나 만들자고 40초를 빈 화면(로딩바)에서 기다리는 유저는 없습니다. 이탈률이 극단적으로 높아질 것이며, "기다림을 상쇄할 재미있는 로딩 화면"을 만드는 것조차 MVP 기간에는 사치스러운 리소스 낭비입니다.

## 3. 예산 및 인프라 비용 초과 (Financial Risk)
해커톤이나 아카데미 프로젝트의 예산은 매우 제한적입니다.

*   **비판 지점:** 외부 API(OpenAI, Stable Diffusion, TTS 등) 호출 비용과, FFmpeg 인코딩을 버텨낼 백엔드 서버(최소 어느 정도 연산 능력이 있는 인스턴스) 비용이 고려되지 않았습니다.
*   **리스크:** 사용자 100명이 테스트로 3번씩만 생성해도 API 비용과 서버 부하가 급증합니다. 자칫하면 발표 당일에 크레딧이 고갈되어 서비스가 멈추는 대참사가 발생할 수 있습니다.

## 4. '무해한 SNS'의 역설 (Moderation 누락)
소셜 기능에서 "고정 이모지만 쓰니까 안전하다"는 것은 1차원적인 방어입니다.

*   **비판 지점:** 악의적인 사용자가 혐오스러운 사진, 기괴한 물체, 성적인 이미지 등을 업로드했을 때, CV/GenAI 파이프라인이 이를 필터링하지 못하고 그대로 "포켓 도감(광장)"에 전시해 버릴 수 있습니다.
*   **리스크:** '안전한 소통'을 표방한 플랫폼에서 부적절한 이미지가 광장에 노출되는 순간, 심리적으로 취약한 타겟 유저들은 더 큰 상처를 받고 서비스의 존재 가치가 부정됩니다. **입력 이미지 필터링(NSFW Check) 파이프라인이 완전히 누락**되어 있습니다.

## 5. 리텐션(Retention) 부재: "그래서 내일 또 올까?"
*   **비판 지점:** 도감에 내 아바타를 올리고, 남의 아바타에 하트를 누르는 경험은 최초 1회(약 3~5분)는 신기하고 재미있습니다. 하지만 그게 끝입니다. 다음 날 사용자가 이 앱을 다시 켜야 할 이유(Core Loop)가 기획에 없습니다.
*   **리스크:** 일회성 '장난감(Toy)'으로 전락할 확률이 높으며, 진정한 의미의 'SNS(네트워크)'나 지속 가능한 서비스로 평가받기 어렵습니다.

## 💡 PM의 긴급 수정 권고 (Actionable Pivot)

위 리스크를 피하면서 3주 안에 성공하려면 다음과 같은 뼈아픈 타협(Pivot)이 필요합니다.

1.  **FFmpeg 서버 렌더링 폐기 $\rightarrow$ 프론트엔드 렌더링으로 전환:** 백엔드에서 MP4 영상을 만들지 마십시오. 프론트엔드(React/Next.js) 단에서 CSS 애니메이션과 HTML5 Audio 태그를 이용해 이미지와 자막을 **동시에 재생(재생 시점에 결합)**하는 방식으로 가야 레이턴시와 서버 비용을 획기적으로 줄일 수 있습니다.
2.  **모더레이션(Moderation) 필수 추가:** 얼굴 검출(CV) 단계에서 사람 얼굴이 아니거나, 외부 API의 NSFW(유해 콘텐츠) 필터링에 걸리면 아예 생성을 차단하는 로직을 1순위로 넣어야 합니다.
3.  **비동기 처리(Task Queue) 아키텍처 필수 도입:** 사용자가 요청하면 즉시 '생성 중' 페이지로 넘기고, 백엔드에서는 순차적으로 작업을 처리한 후 웹소켓(WebSocket)이나 폴링(Polling)으로 프론트엔드에 완료를 알리는 구조로 개발해야 서버가 터지지 않습니다.
